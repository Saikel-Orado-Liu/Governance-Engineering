# ADR-006: CommonMark 合规性测试套件

## 状态
已接受

## 背景
需要对 Markdown 解析管道进行系统性 CommonMark 规范合规性验证。过去的测试仅覆盖零散功能点，缺少面向规范的回归保护。CommonMark 规范包含大量边缘情况（tab 缩进、fence 长度比较、heading 空格要求），手写测试用例容易遗漏。

同时需要追踪已知缺口（Known Gaps）：当前 TokenType 枚举缺少 Setext heading、Blockquote、Ordered list 的专属类型，导致部分 CommonMark 结构只能通过 PARAGRAPH fallback 处理。

## 决策

### 1. JSON 测试用例格式
使用 `{ description, section, markdown, html }` 四字段 JSON 数组格式存储测试用例，存放于 `tests/specs/commonmark-specs.json`。理由：JSON 可由非开发者独立维护（无需编辑 TypeScript），且可作为规范的机器可读副产品供外部工具消费。

### 2. 动态 vitest 运行器
`tests/commonmark-compliance.test.ts` 从 JSON 文件读取用例，按 section 分组动态生成 vitest `describe`/`it` 块。每个用例独立测试，通过/失败由 vitest 原生报告。`afterAll` 中输出 stdout 合规报告：

```
  §4.2: 8/10 (80.0%)
  §4.4: 3/3 (100.0%)
  TOTAL: 25/31 (80.6%)
```

### 3. 已知缺口追踪（IR-001）
合规报告的失败用例中，部分源于已知缺口（Setext heading TokenType 缺失、Blockquote TokenType 缺失、Ordered list TokenType 缺失）。这些缺口记录在模块卡片 `lexer.yaml` 和 `ast.yaml` 的 `known_issues` 字段中，不作为阻止构建的错误处理。目的是在逐步消除缺口时避免遗忘。

## 后果

### 正面
- 新增 CommonMark 规范测试用例只需编辑 JSON，无需修改测试代码
- 合规报告百分比直接衡量规范覆盖进度（量化指标）
- 动态分组测试避免手工维护 describe/it 样板
- 已知缺口明确追踪，修复目标清晰

### 负面
- JSON 格式不支持动态生成测试用例（所有用例静态定义）
- 合规测试依赖 lexer+AST+renderer 端到端管道，不提供单层精度的失败定位
- 当前仅覆盖 §4.2/4.4/4.5/4.8/4.9/5.2，其他章节需逐步补充用例
