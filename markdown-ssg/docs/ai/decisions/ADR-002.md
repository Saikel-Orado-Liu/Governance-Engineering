# ADR-002: AST 解耦架构

## 状态
已接受

## 背景
已完成两阶段词法分析（ADR-001），词法分析器输出扁平的 Token[] 数组。但下游渲染逻辑需要层级结构（如标题下的嵌套段落、列表中的多层级项目）。直接在 Token[] 上做 HTML 渲染会导致大量临时状态管理和重复遍历，代码可维护性差。

## 决策
在词法分析器和渲染器之间插入一个显式的 AST（抽象语法树）层：

1. **Lexer → Token[]**：两阶段词法分析（不变）
2. **buildAST(Token[]) → DocumentNode**：将扁平 Token 转换为层级 AST
3. **renderToHTML(ASTNode) → string**：递归遍历 AST，每个节点类型映射到对应 HTML 标签

架构管道：
```
输入文本 → tokenize → Token[] → buildAST → DocumentNode → renderToHTML → HTML
```

AST 节点由 15 种 discriminated union 组成（BlockNode | InlineNode），renderer 通过 switch 匹配节点类型分发到具体渲染逻辑。

## 后果

### 正面
- **接口边界清晰**：renderer 只依赖 AST 类型，不感知 Token 格式。修改词法分析不影响渲染。
- **递归天然适配层级**：Markdown 的嵌套结构（列表内的段落、段落内的粗体）直接映射到 AST 递归。
- **可测试性提升**：可以单独测试 AST 构建和 HTML 渲染。
- **易扩展**：新增语法块只需添加 AST 节点类型 + 渲染分支，不需修改现有逻辑。

### 负面
- 多一层抽象和一次遍历开销（AST 构建是 O(n) 扫描，对 Markdown 文件可忽略）。
- 部分内联语法的嵌套（如链接内的粗体）需要 AST 节点递归设计得更通用。
