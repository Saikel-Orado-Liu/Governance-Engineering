# ADR-005: Frontmatter 解析与布局回退链

## 状态
已接受

## 背景
需要支持每个 Markdown 文件通过 frontmatter 元数据指定自己的布局模板（layout），
同时保留 CLI --template/--layout 参数的全局覆盖能力。项目已有的模板系统
（DEFAULT_TEMPLATE + renderTemplate）需要扩展以支持 per-file 布局选择，
以及布局名称的路径安全校验。

## 决策

### 1. Frontmatter 解析器（parseFrontmatter）
- 仅支持 `key: value` 格式（`key:` 后必须跟一个空格），不做完整 YAML 解析
- 非 `key: value` 行静默忽略
- frontmatter 必须在文件开头（`---` 是 content 的前 4 个字符）
- closing `---` 缺失时整体回退为 null（整个 content 视为 body）
- 空 frontmatter（`---` 后紧跟 `---`）返回 `{}` 而非 null

### 2. 布局回退链
显式模板/布局参数优先于 frontmatter 元数据：

```
--template (CLI 显式文件路径) → 最高优先级，完全跳过后续回退
→ --layout (CLI 显式布局名) → 仅当 --template 未提供时生效
→ frontmatter.layout (文件内元数据) → 仅当 CLI 未指定任何模板/布局时生效
→ DEFAULT_TEMPLATE (内置默认模板) → 最终保底
```

关键语义：CLI 参数（--template 或 --layout）不仅覆盖 frontmatter，
而且**完全压制** frontmatter.layout——当 CLI 指定了模板时，
文件内的 layout 字段不会被读取。

### 3. 布局路径安全
布局名通过 `/^[\w-]+$/` 正则校验（仅允许字母数字、连字符、下划线），
拒绝路径分隔符和 `..`。最终路径为 `process.cwd()/layouts/{name}.html`。

这与 --template 的白名单校验策略不同（后者校验解析后的绝对路径是否
以项目目录为前缀），因为布局名是从 frontmatter 元数据或 CLI --layout
标志传入的短名称（如 "blog"、"docs"），不是文件路径。

## 后果

### 正面
- 用户可在文件级指定布局模板，无需全局指定 --template
- CLI 参数始终拥有最高优先级，符合 CLI 惯例（显式 > 隐式）
- 布局路径正则校验防止通过 frontmatter 元数据发起路径遍历攻击
- parseFrontmatter 保持零运行时依赖（不引入 YAML 解析库）

### 负面
- parseFrontmatter 不识别完整 YAML 语法，可能误解析复杂 frontmatter
  （如冒号出现在值内、多行值、列表语法），这些情况下的行为是静默忽略
- frontmatter.layout 在 CLI 指定模板时被完全忽略，可能让用户困惑于
  "为什么文件内的 layout 没生效"——需在文档中明确优先级
- 布局文件固定从 `process.cwd()/layouts/` 目录读取，不支持自定义布局目录
