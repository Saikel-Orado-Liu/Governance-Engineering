# ADR-004: 双 CLI 入口架构 — 目录模式与单文件模式共存

## 状态
已接受

## 背景
项目初期只有 src/cli.ts 作为 CLI 入口，提供 `build`（目录=>目录批量转换）和 `serve`（开发服务器）两个子命令。随着项目演进，需要为"用户只想转换一个 .md 文件"的常见场景提供更简洁的 CLI 体验，避免强制用户创建 src/ 和 out/ 目录结构。

需要决定：是在 cli.ts 中追加单文件命令，还是新建独立的 CLI 入口。

## 决策
选择新建独立的 CLI 入口于 src/index.ts，形成双 CLI 入口共存的架构：

1. **src/index.ts（单文件模式 + 库导出）**
   - 同时担任库导出（export type/function）和可执行 CLI 角色
   - 命令格式：`markdown-ssg <input.md> [--output <path>] [--watch]`
   - `bin` 配置指向 `dist/index.js`
   - 使用 DEFAULT_TEMPLATE，不支持自定义模板（保持接口简洁）

2. **src/cli.ts（目录模式，保持不变）**
   - 命令格式：`node dist/cli.js build <src> <out> [--template|--css]` / `node dist/cli.js serve <src> <out> [--port]`
   - 无 bin 注册，需直接调用
   - 支持自定义模板和 CSS 注入

### 选择理由
- **接口隔离**：两种模式的参数解析逻辑完全不同（文件路径 vs 目录路径+子命令），混在一起会破坏单文件模式的简洁性
- **向后兼容**：cli.ts 不需要为单文件模式做适配，已有消费者不受影响
- **自包含性**：index.ts 作为库入口，自然适合同时提供单文件转换的便捷 API 和编程式调用接口

### 注意事项
- 两入口共享相同的底层管道（lexer→ast→renderer→template），但各自独立导入，存在代码重复
- 两个入口均使用 `process.argv[1]` 运行时检测条件执行（ESM 替代 require.main === module 的模式）
- 由于 bin 已从 `md-ssg` 改为 `markdown-ssg`，旧命令 `md-ssg build` 不再可用

## 后果

### 正面
- 用户获得两种工作流：快速单文件转换和完整目录构建/服务
- 库使用者可单独导入 convertFile() 做编程式单文件转换
- cli.ts 不受单文件变更影响，降低回归风险

### 负面
- 需维护两个入口的参数解析和 CLI 行为
- 共享管道代码有两份拷贝（index.ts 和 cli.ts 各自独立导入 lexer→ast→renderer→template），变更需同步两处
- `markdown-ssg build` 在 index.ts 入口下会误将 "build" 当作文件路径而非子命令，使用时需注意入口选择
