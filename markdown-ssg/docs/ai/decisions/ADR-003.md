# ADR-003: 解析器薄代理层设计

## 状态
已接受

## 背景
AST 构建逻辑全部实现在 buildAST（src/ast.ts）中。新增的 src/parser.ts 模块旨在提供清晰的公开 API 入口。需要决定解析逻辑的归属：是逐步将逻辑迁移到 parser.ts，还是让 parser.ts 保持为 buildAST 的薄代理层。

## 决策
选择 parser.ts 作为 buildAST 的薄代理层：

1. **parser.ts 不含新解析逻辑**，仅导出 `parse(tokens: Token[]): DocumentNode`，直接委托 buildAST
2. **所有功能在 buildAST 中实现**：段落合并（mergeParagraphNodes）、CodeBlock.isValid 标记、内联解析等全部保留在 ast.ts
3. **parser.ts 作为公共 API 门面**，未来可在不修改 ast.ts 的前提下，在 parser.ts 中增加预处理/后处理逻辑

### 选择理由
- **避免循环依赖**：将逻辑移至 parser.ts 会导致 parser.ts → ast.ts 解析函数 → parser.ts 的潜在循环
- **现有消费者自动受益**：直接调用 buildAST 的 renderer.ts、cli.ts 等无需改为调用 parse()
- **最小化变更面**：新增 parser.ts 不影响现有模块间的契约

## 后果

### 正面
- 避免双向模块依赖
- 现有消费者不受影响，无需修改既有调用代码
- 未来扩展点清晰：parser.ts 可在不修改 ast.ts 的前提下添加令牌预处理、后处理校验等

### 负面
- 多一层函数调用（微乎其微的性能开销，可忽略）
- parser.ts 当前职责不明确（纯代理），阅读代码的开发者需了解设计背景
