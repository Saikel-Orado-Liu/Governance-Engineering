# ADR-001: 两阶段词法分析

## 状态
已接受

## 背景
Markdown 解析需要同时处理块级结构（标题、段落、代码块、列表）和内联语法（粗体、斜体、行内代码、链接、图片）。如果在一个循环中同时处理两类语法，解析器逻辑会变得复杂且难以测试。需要决定词法分析的架构方式。

## 决策
采用两阶段解析架构：

1. **Phase 1（行级）**：按行扫描输入，识别块级结构（HEADING, PARAGRAPH, BLANK_LINE, CODE_BLOCK, UNORDERED_LIST）。每行/块生成一个块级 Token。
2. **Phase 2（行内）**：仅对 Phase 1 输出的 PARAGRAPH、HEADING、UNORDERED_LIST token 的 raw 文本行，运行正则匹配提取内联 token（BOLD, ITALIC, CODE_INLINE, LINK, IMAGE）。
3. 两阶段结果合并为一组扁平 Token[] 数组，按 (line asc, column asc, raw.length desc) 排序。

## 后果

### 正面
- 职责分离清晰：行级逻辑与行内逻辑解耦
- 容易测试：可以单独测试行级解析和行内解析
- 扩展方便：新增块级语法只需修改 Phase 1，新增内联语法只需在 Phase 2 的 parseInline 中添加 regex 规则
- Token 排序使得下游消费者可以按位置顺序处理

### 负面
- 两遍扫描产生额外的数组遍历开销（但 Markdown 文件通常不大，可忽略）
- 行内解析完全依赖正则匹配，对复杂嵌套语法（如链接内的粗体）支持有限
- 行级解析目前是每行一个 token，没有合并连续段落（这在后续解析阶段处理）
