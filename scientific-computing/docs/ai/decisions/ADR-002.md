# ADR-002: 优化模块采用 ABC 基类策略模式

## 状态

已接受

## 背景

scicomp.optimize 模块需要支持多种无约束优化算法（梯度下降、牛顿法），且未来可能扩展更多算法（共轭梯度法、BFGS 等）。在 API 设计上有两种主流选择：

1. **函数式 API**（如 scipy.optimize.minimize）：统一入口函数 + method 参数选择具体算法，参数通过字典传递
2. **面向对象 API**（基类 + 子类）：每个算法是一个类实例，共享行为通过继承复用，参数通过构造函数传递

两种方案在表达力和扩展性方面存在本质差异。

## 决策

采用 Minimizer ABC 基类 + 具体子类（GradientDescent、NewtonMethod）的面向对象策略模式。

理由：
1. **共享行为集中管理**：收敛检测（_check_convergence 基于梯度无穷范数）、最大迭代终止（_max_iter_reached）等公共逻辑在基类一次实现，所有子类自动获得，消除重复代码
2. **新增算法成本低**：只需继承 Minimizer 并实现 minimize 抽象方法，不需要修改现有代码或选择路由逻辑——符合开闭原则
3. **状态保持自然**：算法超参数（max_iter、tol、step_size）通过构造函数传递并作为实例属性保持，不像函数式 API 需用 dict 或 kwargs 传递
4. **结果类型统一**：所有 minimize 方法返回 OptimizeResult dataclass，接口签名一致，调用方可统一处理
5. **类型系统保障**：Python ABC + @abstractmethod 确保子类必须实现 minimize，否则实例化时报 TypeError，比运行期 method 字符串校验更早发现遗漏

## 后果

正面：
- 添加新优化算法只需继承 Minimizer + 实现 minimize，扩展路径清晰
- 收敛检测等公共逻辑在基类集中维护，修改收敛条件一处生效
- 类型注解精确，IDE 和 mypy 可提供更准确的自动补全和静态检查

负面：
- 比函数式 API 多一层对象实例化开销，简单场景（仅用一个算法）需要 `GradientDescent()` + `.minimize()` 两步而非单函数调用
- 不能像 scipy.optimize.minimize 那样通过一个 method 字符串参数在运行期切换算法——切换算法需要手动更换构造的实例
- 如果未来算法数量增长到 10+，ABC 继承层次可能变深，需要引入组合模式或 Mixin 来控制复杂度
