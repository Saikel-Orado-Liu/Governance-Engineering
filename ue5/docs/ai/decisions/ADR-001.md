# ADR-001: FGameSaveManager 选择静态工具类 + FArchive 架构

## 状态

已接受 (2026-05-08)

## 背景

需要为 Test 模块添加游戏存档管理功能（文件级序列化/反序列化），涉及三种架构选型。

### 方案A: 静态工具类 + FArchive 读写

零实例管理开销，与 FMyClass 一致的模块内风格。通过 FArchive 直接操作文件流进行序列化。

### 方案B: 实例化类 + 内部状态管理

需要实例化后使用，可管理内部状态（如当前存档路径、缓存上下文），但增加了调用方
管理生命周期的负担，与模块现有纯静态风格不一致。

### 方案C: IFileHandle 代替 FArchive

使用 IFileHandle 低级文件操作代替 FArchive，更接近操作系统 I/O 层，
但失去了 FArchive 的 operator<< 序列化便利性和 UE 序列化惯例如自动处理字节序等。

## 决策

选择方案A — 静态工具类 + FArchive。

理由：
1. 与模块现有 FMyClass 风格一致，开发者无需学习新的调用模式
2. FArchive 提供 operator<< 序列化便利，适合存档数据的结构化读写
3. 零实例管理成本，调用方直接 FGameSaveManager::Save() / Load() 语法
4. 当前需求不涉及多存档上下文或缓存状态管理，静态类足够

## 后果

### 正面

- 一致的模块编码风格（与 FMyClass 一致）
- 调用简单，无生命周期管理
- FArchive 序列化便利性（operator<<、自动字节序处理）
- 低心智负担：无实例化、无状态同步问题

### 负面

- 无法管理内部状态（如存档路径缓存、多存档切换），但当前需求无此需要
- 扩展为有状态设计时需要重构为实例化类（方案B）
- 失去 IFileHandle 的底层控制能力（如异步 I/O），但当前同步存档场景不需要

## 替代方案摘要

- **方案B** 更适合需要缓存、多存档上下文管理或事务性写入的场景，当前单一存档需求不需要
- **方案C** 提供更低层次的控制和异步 I/O 潜力，但失去 FArchive 的序列化便利，
  且与模块整体"使用引擎高阶 API"的编码习惯不符

## 与模块现有风格的关系

本决策延续了 FMyClass 确立的"静态工具类"模式，将 Test 模块中该类风格从 1 个类
扩展为 2 个类。两个类在架构风格上保持一致，调用方遵循统一的模式。
