# 治理工程 — 理论与设计

——从 Harness Engineering 到管理学范式的 AI 软件开发体系

> **文档系列说明**：本文为系列第 1 部分（学术文档），聚焦理论基础、设计原则和 Harness Engineering 关系定位。第 2 部分（参考示例文档）为基于 UE5 项目的完整实践落地指南。

---

## 摘要

本文提出"**治理工程**"（Governance Engineering）——一种以管理学原理为理论基础的 AI 开发方法论。
**核心论点**：AI 在软件开发中暴露的质量问题与人类团队面临的管理问题存在结构同构性，因此管理学中经过验证的组织设计原则（专业化分工、标准化流程、制度化知识、需求明确化、分层审查）可以且应该被映射到 AI 工作流设计中。

本文系统对比了现有方法论的演进脉络，建立了五大设计原则与三层架构模型，定义了完整的 10 类专业化 Agent 角色矩阵，建立了 Token 经济学分析框架，并阐明与 Harness Engineering 的互补分层关系。

> [!TIP]
> **如何使用本文档**
>
> - **新接触 AI 开发的团队**：从头通读，重点关注第四章五大设计原则
> - **已有 AI 开发经验的团队**：直接跳转第五章架构模型，第七章 Harness Engineering 关系定位
> - **关注成本优化的团队**：第六章 Token 经济学是必读内容
> - **需要实践落地**：参见系列第 2 部分《UE5 项目参考实现》

## 一、引言：AI 辅助软件开发的核心矛盾

AI 辅助软件开发的根本困境可以用一句话概括：**AI 模型的输出本质是概率性的，而软件工程要求输出必须是确定性的。**

这一矛盾并非 AI 领域独有——人类大脑的输出同样是概率性的。软件工程在过去五十年中发展的核心主题，正是**如何在由不确定性个体组成的系统中，可靠地生产确定性的软件产物**。

因此，当我们审视 AI 辅助软件开发的方法论演进时，有理由期待：被人类软件工程验证过的管理学原则，将在 AI 上下文中找到对应的映射。本文的目的，就是系统地论证这一映射并构建可操作的方法论。

## 二、方法论演进：系统比较与分析

### 2.1 Prompt Engineering 与 Vibe Coding

**Prompt Engineering** 的核心假设是：AI 输出质量由提示词质量决定。实践者通过设计提示词模板、角色设定、思维链等技术引导模型输出。**Vibe Coding** 是其退化形式——反复调整提示词直到"看起来能用"，缺乏系统化验证。

**根本局限**：质量保障完全依赖个人经验和临场判断，不可标准化、不可复现、不可审计——本质是"手艺"而非"工程"。

### 2.2 Spec Coding：需求侧的规范化

**Spec Coding** 主张用结构化的规格说明替代自由文本提示，包括用户故事、验收条件、输入/输出接口契约等。

**贡献**：首次将"需求质量"纳入方法论视野。**根本局限**：隐含假设"只要需求明确，AI 就能产出正确实现"——这在软件工程史上已被反复证伪。Spec Coding 只解决了"输入侧"问题，"过程侧"和"输出侧"的质量保障依然缺失。

### 2.3 Context Engineering：信息环境的优化

**Context Engineering** 认为 AI 的上下文窗口类似人类的"工作记忆"——窗口中可用的信息质量直接决定输出质量。核心实践包括精心选择注入的文件片段、维护 `CLAUDE.md` 等持久化指令、使用 RAG 动态拉取相关代码。

**根本局限**：上下文窗口是有限资源，信息筛选本身就可能出错。更关键的是，它以**单次任务**为优化单元，无法解决跨任务的一致性问题——不同任务即使使用了相似的上下文策略，产出的代码风格、架构决策仍可能不一致。

### 2.4 Harness Engineering：工具层约束的突破

> [!IMPORTANT]
> **Harness Engineering** 代表了当前方法论的最高水平。它不再只是"告诉 AI 该怎么做"，而是通过工具层的硬约束来**限制 AI 能做什么**。

| 机制               | 说明                                                                                   |
| ------------------ | -------------------------------------------------------------------------------------- |
| **Hooks**          | 在关键节点（文件写入前、命令执行前、提交前）插入自动化检查，不符合标准的操作被直接拦截 |
| **权限控制**       | 精细控制 AI 可执行的命令、可访问的文件路径                                             |
| **子 Agent 系统**  | 将任务分派给具有不同权限的子 Agent，实现职责隔离                                       |
| **自动化验证管道** | linting → 编译 → 测试 → 安全扫描，形成自动化质量闸门                                   |

**根本局限**：Harness Engineering 是**机制框架**，而非**设计方法论**。它提供了强大的工具来构建约束，但它不告诉你：应该设置哪些 hooks？应该将工作拆分成几个子 Agent？知识库应该如何组织？审查流程需要几层？

> [!TIP]
> **类比**
>
> Harness Engineering 是 CI/CD 基础设施——它能运行你配置的任何检查；但应该配置哪些检查、开发流程应该如何设计，这些是工程文化和组织设计的问题，不是工具能回答的。

### 2.5 系统性对比

| 维度           | Prompt Engineering | Spec Coding      | Context Engineering | Harness Engineering | **治理工程**       |
| -------------- | ------------------ | ---------------- | ------------------- | ------------------- | ------------------------ |
| **方法论层级** | 输入层             | 输入层           | 输入层              | 机制层              | **设计层**               |
| **核心问题**   | 如何写好提示词     | 如何写清需求     | 如何提供精准上下文  | 如何用工具约束 AI   | **如何设计 AI 组织架构** |
| **质量保障**   | 依赖个人经验       | 依赖需求文档质量 | 依赖信息筛选质量    | 依赖自动化检查      | **依赖系统性组织设计**   |
| **约束性质**   | 建议性             | 建议性           | 建议性              | 强制性              | **设计性**               |
| **根本局限**   | 手艺而非工程       | 只解决输入侧     | 窗口有限            | 有机制无设计        | **组织维护成本**         |

**核心定位**：治理工程不替代 Harness Engineering，而是填补其留下的设计空白——**为"应该构建什么样的约束系统"提供原则和决策依据。**

## 三、理论重构：为什么是管理学？

### 3.1 AI 问题与管理学问题的结构同构性

> [!IMPORTANT]
> **同构性命题**
>
> AI 在软件开发中暴露的各类质量问题，与人类软件团队在缺乏有效管理时暴露的问题，在结构上是同构的——两者共享相同的底层约束：有限理性的智能体、不完整的信息、需要协调多个认知单元完成复杂任务。

| AI 开发中的问题  | 人类团队中的同构问题       | 同构根源                         |
| ---------------- | -------------------------- | -------------------------------- |
| 代码风格不一致   | 团队无编码规范             | 多个执行单元缺乏统一的输出标准   |
| 重复造轮子       | 新员工不了解已有模块       | 执行单元缺乏对系统全局状态的认知 |
| 隐蔽的逻辑 Bug   | 无 Code Review             | 单个执行单元的自检能力有天然上限 |
| 产出偏离需求     | 需求文档模糊               | 信息在传递过程中发生衰减和变形   |
| 长上下文质量衰减 | 单人同时处理过多信息       | 工作记忆是有限资源               |
| 跨会话行为不一致 | 不同员工做类似任务结果迥异 | 缺乏制度化的知识和流程           |

同构性意味着：**管理学在过去一个世纪中发展出的应对这些结构性问题的解决方案，在逻辑上可以迁移到 AI 开发领域。**

### 3.2 放大效应与康威定律推论

> [!WARNING]
> **放大效应**
>
> AI 的高效性使得相同的结构性缺陷产生更大的破坏——一个人类程序员在没有代码规范的情况下，一天可能写出 200 行风格不一致的代码；AI 在同样条件下，几分钟内就能产生 2000 行。**管理流程的价值与 AI 的速度同比放大**，而非相反。

**康威定律的 AI 推论**：AI 代理系统产出的代码结构，将复制该 AI 系统内部的知识结构和通信结构。一个由单一 Agent 运行于单一长上下文的系统，产出将天然倾向于单块式、高耦合——因为生成过程中没有"组织边界"的概念。反之，由多个职责明确的子 Agent 组成的系统，产出将天然倾向于模块化、低耦合。**AI 系统的组织架构不仅是管理问题，它直接塑造产出代码的架构质量。**

## 四、治理工程的五大设计原则

### 4.1 原则一：专业化分工

**管理学基础**：专业化减少个体需要处理的信息量，从而提升在特定领域内的决策质量。

**AI 映射**：Agent 的上下文窗口是稀缺资源。当一个 Agent 同时需要理解需求、设计架构、编写代码、编写测试、审查质量时，上下文被五类不同性质的信息瓜分，每类信息的深度都受到严重制约。将任务按专业领域拆分，等同于为每个 Agent 扩展了有效上下文深度。

**实践准则**：

| 准则                                     | 说明                              |
| ---------------------------------------- | --------------------------------- |
| 一个 Agent 只承担一种角色                | 生成/审查/规划/文档等角色明确分离 |
| 生成 Agent 不被告知审查标准              | 避免"应试式"产出                  |
| 审查 Agent 不被告知生成 Agent 的设计意图 | 确保独立判断                      |
| Orchestrator 不参与具体工作              | 调度者是"管理者"，不是"执行者"    |

### 4.2 原则二：标准化工作流

**管理学基础**：标准化的价值不在于约束创造力，而在于**消除不需要创造力的环节中的变异**，将认知资源集中在真正需要判断的环节。

**AI 映射**：AI 的"创造力"体现在代码生成和问题求解；在需求确认、规范检查、测试验证等环节，需要的是**一致性**而非创造力。标准化的阶段闸门（Stage Gate）确保 AI 在每个阶段完成规定动作后才进入下一阶段，防止跳过关键步骤。

**实践准则**：定义不可跳过的阶段序列（需求复述 → 上下文加载 → 代码生成 → 自动化验证 → AI 审查 → 人工审批），通过 hooks 强制执行，阶段转换时产生可审计的产物。

### 4.3 原则三：制度化知识

**管理学基础**：成熟组织的标志是将关键知识从"默会状态"转化为"显性状态"，使之独立于任何特定个人。

**AI 映射**：AI Agent 的"知识"存在于系统提示词和上下文窗口——两者均不跨会话保留。如果不设计显性化机制，每次新会话都将从零开始，重复之前的探索和错误。制度化知识层是 AI 组织的"外置长期记忆"，为每个新 Agent 提供全部背景信息，无需通过上下文传递。

> [!TIP]
> **模块卡片制度**：每个功能模块维护一份不超过 200 行的简明文档，包含：模块职责（一段话）、对外接口（公开签名及用法）、内部依赖、设计约束（已知的坑）、最后更新时间。这 200 行不是给人看的——是给下一个 AI Agent 看的，恰好可以作为子 Agent 上下文的一部分被加载，而不会挤占任务空间。

### 4.4 原则四：需求明确化

**管理学基础**：需求不明确是项目失败的首要原因（PMI 统计）。将模糊的用户意图转化为明确的技术规格是需要专业方法的独立任务。

**AI 映射**：当人类给 AI 模糊的需求描述时，AI 默认选择**猜测**而非提问——因为其奖励模型倾向于"给出答案"。

> [!CAUTION]
> **危险的不对称性**：AI 在不确定时倾向于脑补，而脑补内容偏离用户意图的成本远高于在需求阶段澄清的成本。

**实践准则**：需求文档必须包含功能目标、输入/输出规格、边界条件、与现有系统的交互、非功能性约束；AI 在开发前必须用自己的语言复述需求理解并获得确认；发现未覆盖的边缘情况时，流程应强制暂停提问而非自行假设。

### 4.5 原则五：分层审查

**管理学基础**：缺陷修复成本随发现阶段呈指数增长——需求阶段的修复成本为 1 单位，测试阶段为 100 单位，生产环境为 1000 单位（Boehm, 1981）。在尽可能早的阶段发现缺陷是经济理性的。

**AI 映射**：

| 缺陷类型 | 说明                           | 审查策略      |
| -------- | ------------------------------ | ------------- |
| 规范缺陷 | 风格、命名、结构等可由规则检测 | 自动化工具    |
| 逻辑缺陷 | 需要理解代码意图才能判断       | 独立 AI Agent |
| 残余缺陷 | 两者都漏掉的问题               | 人工处理      |

> [!WARNING]
> **信息隔离原则**：审查 Agent 必须不被告知生成 Agent 的设计意图。若审查 Agent 知道了"这段代码想干什么"，它可能无意识地按"代码是否实现了意图"来评判，而遗漏了"这个意图本身是否合理"的深层问题。**信息隔离是独立审查的前提。**

### 5.2 关键设计约束

> [!CAUTION]
> **不可违反的设计约束**
>
> 1. **L0 层不执行 L1 层任务**：管理者不亲自干活
> 2. **L1 层 Agent 之间不直接共享上下文**：信息交换仅通过 L2 层文件系统
> 3. **L2 层的每个知识单元有且仅有一个明确的更新负责人**
> 4. **Agent 上下文加载量不超过角色所需信息量的 120%**：20% 余量用于意外发现
> 5. **Agent 间通信仅通过 L2 文件路径引用**：信息不内联于委派消息中

### 5.3 完整 Agent 角色矩阵

基于五大原则和三层架构模型，一个完整的治理工程系统需要以下 **10 类专业化 Agent**：

| #   | 角色          | 管理层对应   | 触发阶段                     | 权限 |
| --- | ------------- | ------------ | ---------------------------- | ---- |
| 1   | **Init**      | 项目启动团队 | 新项目启动 / 新成员加入      | 读写 |
| 2   | **Confirm**   | 需求分析师   | 需求接收后、开发开始前       | 只读 |
| 3   | **Explore**   | 技术调研     | 任务开始时                   | 只读 |
| 4   | **Plan**      | 架构师       | 需求确认后                   | 只读 |
| 5   | **Generate**  | 开发者       | 计划批准后                   | 读写 |
| 6   | **Simplify**  | 即时质量检查 | 代码生成后（与 Review 并行） | 读写 |
| 7   | **Review**    | 代码审查者   | 代码生成后                   | 只读 |
| 8   | **Summarize** | 文档工程师   | 任务完成后                   | 读写 |
| 9   | **Sync**      | 配置管理     | 检测到人类提交时             | 读写 |
| 10  | **Refactor**  | 技术债务管理 | 定期（每迭代/里程碑）        | 读写 |

## 六、Token 经济学：治理工程架构的成本优化

### 6.1 核心问题

将任务拆分给多个专业化 Agent 引入了额外的**沟通成本**——Orchestrator 委派任务、Agent 返回结果、上下文不可避免地重叠。这是否会导致总 Token 消耗远超单一 Agent 模式？

### 6.2 成本模型

**单一 Agent**：$C_{single} = C_{context} + C_{generation} + C_{error}$

**多 Agent 治理工程**：$C_{org} = C_{orchestrator} + \sum C_{agent_i} + C_{comm} + C_{knowledge}$

看似 $C_{org}$ 天然大于 $C_{single}$，但这忽略了三个关键因素：

> [!IMPORTANT]
> **被低估的隐性成本**
>
> 1. **$C_{error}$ 被严重低估**：单一 Agent 缺乏独立审查，错误在下游才被发现。早期缺陷修复成本仅为后期的 1/10 到 1/1000（Boehm, 1981），审查 Agent 在代码生成后立即拦截问题，节省的 Token 远超审查本身的消耗。
> 2. **$C_{context}$ 随项目规模非线性增长**：单一 Agent 处理大型项目时必须塞入大量无关信息；治理工程架构的每个 Agent 仅加载最小信息集。
> 3. **$C_{comm}$ 可以通过架构设计显著压缩**：沟通成本取决于 Agent 之间接口的"表面积"设计。

### 6.3 核心优化原则：沟通表面积最小化

最核心的优化是将信息传递从"推送模式"（Orchestrator 广播）转为"拉取模式"（Agent 按需获取）：

```text
× 差（宽接口，推送模式）：
  "我们需要在 UInventoryComponent 中添加排序功能，该组件在
   Source/.../InventoryComponent.h，继承自 UActorComponent……"
   → ~200 Token

✓ 优（窄接口，拉取模式）：
  "实现需求文档 REQ-042 中描述的排序功能。
   相关模块索引：docs/ai/modules/inventory.yaml"
   → ~30 Token，Agent 自行读取模块卡片获取上下文
```

### 6.4 五维优化策略

| 策略                      | 说明                                                                                             | 效果                  |
| ------------------------- | ------------------------------------------------------------------------------------------------ | --------------------- |
| **模板化委派**            | Orchestrator 使用标准 YAML 模板委派任务；信息通过文件路径引用，不内联传递                        | 消除委派的 Token 浪费 |
| **结构化输出**            | 子 Agent 返回机器可解析的结构化摘要（状态、文件列表、风险），而非自然语言叙述                    | 减少结果传递消耗      |
| **渐进式上下文加载**      | 先加载 MODULE_INDEX（~50 行）→ 按需加载模块卡片（~200 行/个）→ 必要时才读源码                    | 上下文始终最小化      |
| **任务合并与 Agent 复用** | Must-Read 文件重叠度 > 60% 时合并到同一 Agent；已加载上下文的 Agent 优先复用                     | 消除重复加载成本      |
| **Token 预算制度**        | 为每个 Agent 角色设定上下文预算上限（Orchestrator ≤ 20000；Reviewer ≤ 30000；Generator ≤ 50000） | 强迫精准加载          |

> [!WARNING]
> **典型反例**：为一个 50 行工具函数启动 Planner → Generator → Reviewer 三个 Agent——流程开销可能超过实际代码生成消耗。此时应改为：单 Agent 直接生成 + 自动化 lint 检查。

### 6.5 反直觉结论：中等以上任务多 Agent 更省 Token

> [!IMPORTANT]
> **核心结论**：对于复杂度超过一定阈值的任务，治理工程多 Agent 架构的**总 Token 消耗低于**单一 Agent 模式。单一 Agent 在长对话中存在上下文膨胀、错误修正成本高、缺乏"组织记忆"导致反复描述背景等系统性成本黑洞。

**经验数据估算**（中等复杂度 UE 功能模块，方向正确，非受控实验数据）：

| 成本项     | 单一 Agent                    | 治理工程架构                             |
| ---------- | ----------------------------- | -------------------------------------- |
| 上下文加载 | 40000 Token（全量）           | 15000 Token（精准，各 Agent 去重合计） |
| 代码生成   | 30000 Token                   | 32000 Token（含审查后微调）            |
| 错误修正   | 20000 Token（隐性，常被忽略） | 5000 Token（审查前置捕获）             |
| 通信开销   | 0                             | 6000 Token                             |
| **合计**   | **~90000 Token**              | **~58000 Token（-36%）**               |

## 七、与 Harness Engineering 的关系定位

本文的方法论与 Harness Engineering 的关系并非替代或竞争，而是**互补分层**：

| 层面         | Harness Engineering                 | 治理工程                 |
| ------------ | ----------------------------------- | ------------------------------ |
| **关注点**   | 如何约束 AI 的行为                  | 应该构建什么样的约束体系       |
| **输出物**   | 工具配置（hooks、权限、工作流定义） | 设计原则与架构决策             |
| **类比**     | CI/CD 基础设施与工具                | 工程文化与开发方法论           |
| **问题类型** | "如何实现"（Implementation）        | "为何如此设计"（Architecture） |
| **可迁移性** | 工具绑定的具体配置                  | 跨工具、跨平台的设计原则       |

> [!TIP]
> **协同关系**
>
> 治理工程定义了"应该做什么"，Harness Engineering 提供了"怎么做"的手段。具体而言：
>
> - 五大原则 → 推导出需要的 hooks 类型和触发条件
> - 三层架构模型 → 指导 Agent 系统的角色划分和权限配置
> - 制度化知识原则 → 驱动 Memory 机制的组织方式

> [!WARNING]
> **边界情况**
>
> Harness Engineering 能够实施任何设计决策，无论是好的还是坏的。如果治理工程的设计本身存在缺陷（如模块划分不合理、审查标准过于宽松），Harness Engineering 会忠实地、高效地执行这些有缺陷的设计——它将放大好的设计，也同样放大坏的设计。这是为什么设计层必须独立于机制层被审慎论证。

### 7.1 实践映射：五大原则 → Harness Engineering 配置

| 治理工程原则 | Harness Engineering 映射                                       |
| -------------- | -------------------------------------------------------------- |
| 专业化分工     | 子 Agent 系统 → 不同 Agent 分配不同权限和工具集                |
| 标准化工作流   | Hooks 系统 → 在阶段闸门处插入自动化检查，强制流转              |
| 制度化知识     | Memory 系统 → 文件化知识库 + 共享 agent-memory                 |
| 需求明确化     | Confirm Agent → 需求确认闸门，歧义检测 + 理解复述              |
| 分层审查       | Review Agent + Hooks → 自动化检查 → AI 审查 → 人工审批三层过滤 |

## 八、适应性：何时使用、何时简化

任何方法论都有经济边界。治理工程的设计假设是：

- 项目达到一定规模（模块数 ≥ 5 或代码行数 ≥ 10,000）
- 对质量有较高要求（生产代码，非原型/概念验证）
- 开发是持续性的（非一次性任务）

### 8.1 成本-收益分析

**成本项**：

| 成本项   | 说明                         |
| -------- | ---------------------------- |
| 文档维护 | 模块卡片需随代码变更同步更新 |
| 流程延迟 | 阶段闸门增加了端到端时间     |
| 设计决策 | 需要投入时间思考组织架构本身 |

**收益项**：

| 收益项       | 说明                                            |
| ------------ | ----------------------------------------------- |
| 缺陷提前发现 | 审查层在早期阶段捕获问题（成本低 10-100 倍）    |
| 知识不丢失   | 每次任务的产出被制度化，后续 Agent 无需重新探索 |
| 架构一致性   | 专业化 Agent 在不同任务间保持一致的模块边界     |
| 可审计性     | 每个阶段的产物可追溯、可复盘                    |

### 8.2 适用性判断函数

```python
# 启发式判断函数
def should_use_organizational_architecture(task):
    if task.complexity < "1_hour_human_equivalent":
        return "单一 Agent + 单次对话（够用且高效）"
    elif task.complexity < "1_day_human_equivalent":
        return "简化的两层模型（Orchestrator + 单一执行 Agent + 基本审查）"
    else:
        return "完整的三层治理工程"
```

> [!CAUTION]
> **对抗性警示**
>
> 最大的风险不是方法论的局限，而是**过早过度工程化**——为一个 500 行的工具脚本搭建完整的三层架构，流程开销超过实际开发时间，导致方法论本身成为生产力的障碍。架构的复杂度应该与任务的复杂度匹配，原则的严格程度应该与质量要求匹配。

### 8.3 渐进式采纳路径

```
Level 0: 无架构
  → 单一 Agent + 随意对话

Level 1: 基本文档化（入门门槛最低）
  → CLAUDE.md + 模块卡片 + 编码规范索引
  → 收益：AI 产出的风格一致性显著提升

Level 2: 引入 Confirm + Review（性价比最高的增量）
  → + Confirm Agent（需求确认）+ Review Agent（独立审查）
  → 收益：需求偏离和低级 Bug 大幅减少

Level 3: 完整任务流水线
  → + Plan Agent + Summarize Agent + Simplify Agent
  → 收益：任务可预测性提升，知识开始积累

Level 4: 人机协作闭环
  → + Sync Agent + Refactor Agent
  → 收益：人类与 AI 在同一知识基础上协作，技术债务可控

Level 5: 持续优化
  → + 健康度度量 + 自动化架构设计实验
  → 收益：架构本身开始自我改进
```

> [!TIP]
> **建议路线**
>
> 大多数团队应该从 Level 1 开始，在确认价值后逐步升级到 Level 2 - 3。Level 4 - 5 仅适用于 AI 深度参与日常开发的团队。每次升级前，评估当前级别的成本收益比，确认增量价值超过增量成本后再升级。

## 九、结论与展望

### 9.1 核心贡献

本文提出并系统论证了**治理工程方法论**，其主要贡献包括：

| 贡献                                      | 说明                                                                                                                |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| **揭示结构同构性**                        | AI 开发问题与管理学问题的结构同构性，论证了管理学原则迁移的理论合法性                                               |
| **系统对比现有方法论**                    | Prompt Engineering、Spec Coding、Context Engineering、Harness Engineering 的贡献域和局限边界                        |
| **提出五大设计原则**                      | 每条原则均提供了管理学基础 → AI 映射 → 实践准则的完整论证链                                                         |
| **建立 Token 经济学框架**                 | 论证了多 Agent 架构在中等以上任务中总 Token 消耗反而低于单一 Agent 模式，提出沟通表面积最小化原则                   |
| **定义完整 Agent 角色矩阵**               | 10 类专业化 Agent（Init/Confirm/Explore/Plan/Generate/Review/Summarize/Sync/Refactor/Simplify）的完整定义与调度规则 |
| **建立三层架构模型**                      | 调度层-执行层-知识层的结构定义，每层的职责边界与信息流向                                                            |
| **阐明与 Harness Engineering 的互补关系** | 设计层与机制层的分层协作                                                                                            |

### 9.2 范式转变

本文所倡导的转变可以概括为：

| 旧范式                        | 新范式                               |
| ----------------------------- | ------------------------------------ |
| AI 是一个需要被**使用**的工具 | AI 是一个需要被**管理**的组织        |
| 质量取决于提示词写得多好      | 质量取决于组织架构设计得多好         |
| 优化 AI 的输出                | 优化 AI 系统的结构                   |
| 更好的指令 → 更好的代码       | 更好的流程 + 更好的结构 → 更好的代码 |
| 人的角色是"提问者"            | 人的角色是"架构师"和"管理者"         |

> [!IMPORTANT]
> **心智模型的差异**
>
> 这不是语义游戏。这两种心智模型在每一个实践决策点上导向不同的选择——当面对一段质量不达标的 AI 生成代码时：
>
> - **工具心智**会问"我应该怎么调整提示词来让 AI 写出更好的代码？"
> - **管理心智**会问"我的流程中哪个闸门应该拦截这段代码但没有拦截？"

### 9.3 未来方向

| 方向               | 说明                                                                                             |
| ------------------ | ------------------------------------------------------------------------------------------------ |
| **量化验证**       | 通过受控实验测量分层组织架构相对于单一 Agent 模式在缺陷率、可维护性指标上的量化差异              |
| **自动化架构设计** | 探索由 AI 辅助设计 AI 组织架构的可能性——给定项目特征，自动生成推荐的 Agent 角色划分和 hooks 配置 |
| **跨平台验证**     | 在 Claude Code 之外的工具平台（Cursor、Copilot、Aider 等）中验证治理工程原则的可迁移性     |
| **度量体系**       | 建立 AI 组织健康度的度量指标（如"知识覆盖率"、"审查拦截率"）                                     |

## 核心信条

> [!IMPORTANT]
> **核心信条**
>
> 任何时候你对 AI 的产出不满意，不要问"AI 为什么做不好"，而要问"我的治理工程架构哪里出了漏洞"。
>
> 因为一个设计良好的系统，其输出的质量不应该依赖于任何单个执行单元的能力——无论这个执行单元是人的大脑，还是 AI 的神经网络。这便是工程精神的核心：**用系统性的设计对抗个体的不确定性。**
