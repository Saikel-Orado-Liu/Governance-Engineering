## 快速开始

将模板 `temp` 文件夹中的内容复制到你的项目中：

```bash
cp -r temp/. .
```

然后运行 `/init` 适配到你的技术栈：

```text
/init
```

在编码前对比架构方案：

```text
/optimize <你的方案>
```

里程碑时全量代码健康扫描：

```text
/refactor
```

手动改代码后同步回 AI 知识库：

```text
/sync
```

每个开发任务自动流经 `confirm → plan → develop → inspect → test → commit`。你描述需求，调度者处理其余一切。

## 架构总览

### 为什么需要治理工程

当我们用 AI 编写软件时，我们不断遇到相同类型的失败：**输出不一致、上下文遗忘、职责混乱、未经验证的假设**。这些不是 AI 的问题，而是**管理问题**。

人类软件团队也曾饱受这些问题的困扰——直到我们发明了分工协作、编码规范、代码评审、设计文档和 CI/CD 流水线。这些不是技术发明，而是**组织发明**。

> AI 在软件开发中暴露的质量问题，与人类团队在没有有效管理时暴露的问题，在结构上是**同构的**。

如果这个命题成立，那么解决方案不是更好的提示词或更聪明的模型——而是**更好的治理**。治理工程将管理学中五大核心原则映射到 AI 辅助开发工作流中：

| 管理学原则     | AI 工作流对应                        |
| -------------- | ------------------------------------ |
| **专业化分工** | 范围窄、定义明确的专业化 Agent       |
| **标准化流程** | 强制执行工作流检查点的阶段闸门 Hooks |
| **制度化知识** | 跨会话持久化的共享 Memory 结构       |
| **需求明确化** | 含验收标准的结构化任务定义           |
| **分层审查**   | 关注点分离的多 Agent 审查管道        |

治理工程与 [Harness Engineering](https://openai.com/index/harness-engineering/) 是互补的分层：Harness Engineering 提供约束 AI 的机制，治理工程提供选择哪些约束、为何如此选择的设计理由。

### 架构特性

- 🏛️ **管理学理论基础** — 基于经过验证的组织设计原则，而非临时经验法则
- 🤖 **11 个专业化 Agent** — 每个 Agent 具有窄范围、显式接口和定义的通信协议
- 🔒 **阶段闸门 Hooks** — 自动检查点，防止工作流步骤被跳过
- 🧠 **结构化记忆** — 团队共享的记忆层级，跨会话持久化知识
- 🔌 **技术栈无关** — 方法论适用于任何技术栈；模板可生成特定技术栈的配置
- 📐 **分层架构** — 三层模型（编排层 → 领域层 → 执行层），可扩展复杂度
- 📊 **模型分级费用控制** — 不同 Agent 按任务复杂度分配不同能力模型，将上下文窗口预算作为第一等关注点设计

### 优势

- **降低产出方差。** 结构化工作流产生的 AI 输出比非结构化提示更一致。
- **随复杂度扩展。** 项目中的 Agent 和步骤越多，治理的回报越大。
- **工具无关原则。** 设计原则适用于任何 AI 编码助手。
- **持久化知识。** Memory 结构在会话重启和模型变更后仍然存在。
- **AI 模型分离。** 常规 Skill 方案下所有对话使用统一模型，搜索、确认、同步等所有步骤都会消耗高成本模型。治理工程的 Agent 架构按任务分级：搜索类任务使用轻量模型，代码生成等核心任务才调用高质量模型，大幅降低总体费用。
- **上下文分离。** 子 Agent 独立上下文窗口使主对话 AI 无需承担全部上下文负载，主对话上下文可压缩至 40% 以下而保持最佳输出质量。同时各 Agent 的领域上下文高度聚焦，避免了长上下文中的注意力稀释问题。

### 局限

- **前期投入。** 配置 Agent、Hook 和 Memory 结构需要时间，在编写特性代码之前。
- **不适用于一次性脚本。** 对于单次提示即可完成的任务，治理开销是浪费。
- **学习曲线。** 方法论假设你对组织设计概念有一定了解。
- **Claude Code 偏向。** 当前参考实现针对 Claude Code；适配其他平台需要翻译工作。

## 版本说明

### 当前版本 (v1.0)

当前架构实现了管理学五大原则的基本映射：11 个专业化 Agent、6 个阶段闸门 Hook、5 层结构化 Memory、三层编排模型（编排层 → 领域层 → 执行层）。模板 `temp/` 提供开箱即用的项目配置，配合 `/init` 适配技术栈。

### 当前版本与理想架构的差距

1. **上下文传递开销。** 子 Agent 之间虽有上下文传递，但每个 Agent 仍需重新读取其职责范围内的具体文件，相比传统单会话方法在沟通上消耗了更多上下文。
2. **时间开销。** 工作流中的多 Agent 调度、审查和确认环节比传统方法消耗更多时间。在 DeepSeek V4 API 的实测下，一个高级需求平均耗时 20-40 分钟。
3. **Claude Code 兼容性。** 通过模板创建的架构在 Claude Code 的可视化插件或应用中偶发"卡死"现象——界面无响应但 AI 实际已完成任务，用户需要手动刷新或重启对话。
4. **初始化自动化不足。** `/init` 不会自动查找和安装对应的 MCP 工具，也不判断架构兼容性。在 AI 缺少足够工具支持时表现可能不如预期，需要用户在初始化时或后续主动告知架构所需的外部工具和能力边界。
5. **自主发挥受限。** 每个子 Agent 严格按照既定目标执行任务，不知道原始用户需求的全貌。在需要 AI 完全掌控项目的场景（例如一个不懂编程的人想完整搭建一个漂亮的网页）中，表现可能不如无架构的自由对话。
6. **快速迭代代价过大。** 当前架构更适合中大型项目，对小项目和快速迭代场景而言流程开销过高，投入产出比不理想。

## 核心信条

> 任何时候你对 AI 的产出不满意，**不要问"AI 为什么做不好"，而要问"我的 AI 治理架构哪里出了漏洞"**。因为一个设计良好的系统，其输出的质量不应该依赖于任何单个执行单元的能力。

## 致读者的话

非常感谢每一位阅读并支持本项目的读者。

2025 年 11 月，我正在线上合作开发一系列 SaaS 项目，到 2026 年 2 月，我开始陆续接触完整的 Agent AI 开发（Claude Code）和 Harness Engineering。在线上合作的过程中，我发现了一件事：在需求不明确的情况下，即使是一名优异的开发人员也会做出和项目需求不相称的代码。同时，代码还需要通过测试、Linter 等硬性要求才能发布到非本地的测试开发环境中，才算真正合格。

而 AI 遇到的大多数代码问题，除了上下文不足，更关键的是开发者发送给 AI 的需求本身就不够清楚——而 AI 又倾向于“自我揣测”而非追问用户。这导致 AI 写的代码即使通过了也往往不符合真实需求，同时 AI 也无法完全保证代码风格一致或准确无误。我立刻意识到，至少在产生错误这件事上，AI 和人类本质上是同源的：面对相同的问题，会有完全不可预见、不可控的结果（黑箱操作）。而解决人类这类问题的方法，早在上个世纪就已经被提出来了。

于是，我在 2026 年 4 月底开启了“组织型 AI 架构”的设计，也就是 Governance Engineering。本质上，它是把 AI 设计成一个微型的“代码外包团队”而非"单个代码编辑工具"——AI 内部有完整的流程控制，以确保“一切按照预期执行”。通过可控流程来抵挡不可控的输出变量，同时使用 confirm 模块严格确保 AI 能够正确理解用户需求而不是瞎猜。这与现实中企业在员工能力不均时仍能确保代码质量的逻辑一致：严格的 review、lint、CI 流水线即使面对能力参差不齐的开发者也能守住质量基线。AI 也是一样——即使模型本身质量较低，好的流程也能缩小模型间的表现差距，使其至少接近标准水平。反过来说，如今各大公司的代码库被能力不足的维护者经手后质量肉眼可见地下降，恰恰印证了缺乏流程约束的后果。在测试下，这套流程能够完美执行经过 confirm 输出的最终确定需求；如果输出与预期不符，往往是 confirm 或 plan 的结论不够详细，或者 AI 缺少相关的测试、开发工具所导致。

我不是专业的 AI 从业者或架构师。最早的架构只花了两三天完成，是一个极其简陋的版本。我没有能力也没有资源对整个架构进行实验室级别的质量验证。理想的架构实现，或许需要对整个 Agent AI 体系进行重构才能达成。我十分欢迎各位 AI 大神 Fork 或改进这个项目的架构细节，以用更少的时间和费用完成更高质量的任务。

或许有更早的论文（我确实在项目发布前就见过至少一篇类似的文章）论述了 Governance Engineering 的思想内核，但这确实是我的独立创作。我同时也希望，我的想法能带来一定的学术价值。

—— Saikel Orado Liu
