# ADR-003: 战斗系统架构 — 纯函数 CombatSystem + 状态机 TurnManager

## 状态
已接受

## 背景
FR-03 要求在单位系统和网格地图基础上实现回合制战斗：移动后自动触发战斗、敌方 AI 自动行动、回合交替（player -> enemy -> gameOver）。需要确定战斗逻辑的组织形式（纯函数 vs 类）、回合管理的状态模式、以及敌方行动策略。

## 决策

### 1. CombatSystem 纯函数模块 + TurnManager 状态机分离
- **CombatSystem** 为纯函数模块（非类），导出 `executeCombat(attacker, defender, unitManager) => CombatResult`
- **TurnManager** 为类状态机，管理 `player -> enemy -> gameOver` 状态流转
- 分离理由：CombatSystem 只负责伤害计算和阵亡处理，无状态、无副作用（除 `unitManager.removeUnit` 外）；TurnManager 管理回合流和敌方 AI 编排，各有独立测试职责

### 2. 阵亡单位物理移除（removeUnit）
- 防御方阵亡 -> 立即 `unitManager.removeUnit(defender)`；反杀攻击方 -> 立即 `removeUnit(attacker)`
- 与 ADR-002 的差异：ADR-002 原设计仅通过查询过滤死亡单位（保留尸体在数组中不阻塞 isOccupied），战斗系统引入后需要物理移除以避免 `getUnitAt`/`isOccupied` 误判（战斗轮次中同格可能被多单位查询）
- 查询层（`getAllUnits`/`getUnitsByTeam`）仍保留 `isAlive` 过滤作为二次保障

### 3. 反击机制：仅相邻近战反击
- 防御方存活且与攻击方格相邻（4 方向）时才触发反击
- `DIRECTION_OFFSETS: [-1,0],[1,0],[0,-1],[0,1]` 与 MovementEngine 一致
- 反击使用防御方对攻击方的克制乘数（1.5 / 0.67 / 1.0）

### 4. 敌方 AI：Manhattan 贪心策略
- 距离评估：Manhattan 距离（非 BFS 加权）—— 轻量计算，一回合内全部敌方单位都评估
- 移动策略：贪心单步（4 方向中选择最小化到目标距离的可行格），不走全路径规划
- 攻击目标：相邻敌人中 HP 最低者（非随机、非最近），模拟"优先消灭残血"的战术直觉
- 移动后二次检查：移动后若进入相邻格子，额外触发一次战斗

## 备选方案

| 方案 | 放弃理由 |
|------|---------|
| CombatSystem 类（new CombatSystem().execute()） | 类无内部状态，增加不必要间接层；纯函数对测试更友好（无需 mock 实例、无构造开销） |
| CombatSystem+TurnManager 合并 | 违背 SRP，combat 逻辑和 turn 管理合并不利于后续扩展（如添加 buff/debuff 事件、多队伍回合） |
| 死亡单位保留在数组（仅查询过滤） | ADR-002 原方案，但战斗系统需要腾出网格位置给其他单位移动/攻击后检查；removeUnit + 查询过滤双层保障更安全 |
| 敌方 AI：BFS 全路径规划 | 每回合每个敌方单位跑全网格 BFS 计算量过大（当前最多 3 个敌方单位），贪心单步足够产生合理"逼近"行为 |
| 敌方 AI：A\* 路径规划 | 过度设计——4 方向贪心单步产生自然的"向目标靠近"行为，无需路径缓存或方向优先级 |
| 敌方 AI：随机攻击目标 | 玩家体验差——AI 应优先消灭低威胁/残血目标；HP 最低策略简单且有效，玩家可预测 AI 行为 |

## 影响

- 正面：CombatSystem 纯函数可直接在测试中调用（不依赖 TurnManager 实例），9 个测试覆盖所有战斗路径
- 正面：TurnManager 的 `combatFn` 依赖注入允许测试中 mock 战斗函数，6 个 TurnManager 测试不依赖真实战斗逻辑
- 正面：敌方 AI 贪心策略每单位 O(1) 计算量（仅评估 4 个相邻格），不会随网格大小或单位数量增加
- 正面：Player 回合移动后触发战斗、Enemy 回合循环触发战斗、gameOver 统一检测——回合系统的三条路径均在 TurnManager 中收敛
- 约束：`executeCombat` 调用 `Unit.takeDamage(atkDamage + def)` 采用 reverse-calculation 模式，与 Unit 内部减伤公式耦合（详见 lessons-learned）
- 约束：主流程中（main.ts）用 `setTimeout` 链编排敌方行动，时序依赖于 setTimeout 回调嵌套，而非 async/await 或事件队列
- 约束：同队战斗未做校验，由调用方（main.ts、TurnManager）保证攻击方与防御方不同队
