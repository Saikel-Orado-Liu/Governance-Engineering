# ADR-002: 前端拖拽方案选型（@dnd-kit + 乐观更新 per-task 回滚）

## 状态

已接受

## 背景

Task Board 需要看板视图（Kanban），核心交互是跨三列（todo/in_progress/done）拖拽任务卡片。关键约束：

1. 拖拽必须即时响应用户操作，不等待 PATCH API 返回
2. 失败时自动回滚到正确状态
3. 支持快速连续拖拽多个任务

React DnD 生态有三个主流方案：react-dnd（配置重、HTML5 API 包装繁琐）、react-beautiful-dnd（Atlaskit 已宣布停止维护）、@dnd-kit（模块化、TypeScript 优先、React 18 原生）。

## 决策

1. **拖拽库选型**：采用 @dnd-kit（@dnd-kit/core + @dnd-kit/sortable + @dnd-kit/utilities）。理由：TypeScript 类型推断完整；DndContext + sensors + useSortable 声明式 API；模块化按需引入（~30KB gzip 总计）；社区活跃持续维护。

2. **乐观更新回滚策略**：使用 `useRef<Map<string, Task>>`（per-task Map）而非单一数组 `useRef<Task[]>`。每个 taskId 独立存储拖拽前状态，回滚时只还原失败任务，避免快速连续拖拽中单一 ref 被覆盖导致的竞态。

3. **样式隔离**：CSS Modules 组件级样式（.module.css），零运行时开销。Vite 6 原生编译支持，无需额外配置。

4. **错误处理**：每列独立三态渲染——加载中显示 "Loading..." 文字、空列显示 "No tasks" 占位、失败显示错误消息 + Retry 按钮。非关键操作失败通过 Toast 组件反馈（4 秒自动消失 + useCallback 稳定引用防计时器重置）。

5. **组件层级**：KanbanBoard(DndContext) → KanbanColumn(useDroppable + 三态) → KanbanCard(useSortable + 拖拽手柄)。状态集中管理于 useTasks hook，组件层纯渲染。

## 后果

正面：
- @dnd-kit 声明式 API 将拖拽逻辑与业务渲染解耦，KanbanCard 只关注 useSortable 属性，不关心列逻辑
- per-task Map 回滚在快速连续拖拽场景下行为正确，不会回滚到错误的中间状态
- CSS Modules 零运行时开销，无 className 冲突，Vite 原生支持
- useMemo(分组)/useCallback(onClose) 减少不必要重渲染

负面：
- 增加 ~30KB gzip bundle 体积（三包 @dnd-kit），对首屏加载有轻微影响
- CSS Modules 在动态主题等场景弱于 CSS-in-JS 方案
- per-task Map 在大量任务场景下内存线性增长（每任务一个 ref entry，当前规模可忽略）
