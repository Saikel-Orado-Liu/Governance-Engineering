# ADR-001: SQLite 跨线程连接配置（check_same_thread=False + WAL）

## 状态

已接受

## 背景

FastAPI 使用 async 线程池处理请求，而 Python `sqlite3` 默认禁止跨线程使用同一连接（`check_same_thread=True`）。当多个请求并发到达时，会触发：

```
sqlite3.ProgrammingError: SQLite objects created in a thread can only be used in that same thread
```

此外，默认回滚日志（rollback journal）模式在并发读写场景下容易产生 `database is locked` 错误，因为写操作会阻塞其他读写操作。

## 决策

1. 创建 `sqlite3.Connection` 时设置 `check_same_thread=False`，允许多线程共享同一连接
2. 启用 WAL（Write-Ahead Logging）模式替代默认回滚日志，通过 `sqlite-utils` 的 `enable_wal()` 方法
3. 通过 FastAPI `lifespan` 上下文管理器（`asynccontextmanager`）管理连接生命周期：启动时创建连接，关闭时清理
4. 生产部署限制 Uvicorn `workers=1`，确保单进程单连接模型安全

## 后果

正面：
- 消除跨线程异常，保障多请求并发安全
- WAL 模式显著降低 `database is locked` 概率
- WAL 模式下读写可部分并行，提升吞吐量

负面：
- `check_same_thread=False` 削弱了 sqlite3 的线程安全检查，应用层需保证连接不被多线程同时写入
- WAL 模式在极端崩溃场景下可能遗留 WAL 文件（.wal/.shm），需手动清理
- 不适用于多进程部署（每个进程需独立 DB 连接，或使用其他数据库如 PostgreSQL）
