# ADR-004: 乐观更新回滚 — 深拷贝快照瞬时恢复替代全量网络重载

## 状态

已接受

## 背景

ADR-003 搭建的看板组件使用 `updateTaskStatus().catch(() => load())` 作为拖拽失败的乐观更新回滚策略。load() 会触发 `fetchBoard()` 全量 HTTP 请求，导致：

1. Board 进入 loading skeleton 状态 → UI 闪烁
2. 重新获取已在前端呈现的同一批数据，网络开销浪费
3. 在 3G/弱网环境下闪烁延迟可达 200-500ms

此策略在 ADR-003 中已知并记录为负面后果（"失败时全量 reload 在网络延迟大时可能导致 UI 闪烁"）。

## 决策

1. **快照时机**：在 handleDrop 执行乐观更新**之前**，用 `JSON.parse(JSON.stringify(columns))` 对当前 columns 全量做深拷贝快照，挂载于 `dragState.current.snapshot`
2. **恢复路径**：API 调用（updateTaskStatus/updateTaskReorder）的 `.catch()` 中执行 `setColumns(snapshot)`，将列数据瞬时恢复到拖拽前状态，零网络延迟
3. **load() 保留用途**：useEffect 初始数据加载 + error 重试按钮。不再用于乐观更新回滚
4. **快照范围**：全列 columns（`BoardColumn[]`），而非仅涉列。理由：简单可靠，避免计算涉列边界的复杂性；中小看板（数百卡片以内）深拷贝开销可忽略
5. **深拷贝方案**：`JSON.parse(JSON.stringify())` 对纯数据 Task 对象安全高效（无函数/循环引用/Date/Map/Set），零外部依赖

## 后果

正面：
- 回滚零网络延迟，O(1) 内存赋值操作，UI 无闪烁
- 实现简单，<25 LOC 变更，零外部依赖
- 消除 ADR-003 已知负面后果 #3

负面：
- `JSON.parse(JSON.stringify())` 在大看板（上千卡片）场景下有 O(n) 性能开销和内存占用
- 深拷贝假设 Task 为纯数据对象。若未来 Task 引入非序列化字段（Date/Map/Set/函数），需改用 `structuredClone()` 或 lodash.cloneDeep
- 全量深拷贝冗余：每次拖拽拷贝所有列，而非仅涉列
